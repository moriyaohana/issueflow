# IssueFlow Code Review Findings

Single flat list of concrete, actionable issues from a 5-agent review (spec conformance, requirements coverage, code quality, security, tests). Sources, with full per-agent reports, live in [tmp/review/](tmp/review/). Each bullet is self-contained for the fixer.

## Spec Conformance

- **Problem:** `POST /users/update/:userId` returns the full updated user object, but the README documents an empty 200 response body for this row. **File:** [src/users/users.controller.ts](src/users/users.controller.ts#L47-L56), [README.md](README.md#L64). **Fix:** Return `Promise<void>` from the handler so the response body is empty as documented.
- **Problem:** `POST /auth/logout` returns `{ok: true}` but README documents an empty 200 response body. **File:** [src/auth/auth.controller.ts](src/auth/auth.controller.ts#L35-L43), [README.md](README.md#L72). **Fix:** Return `void` so the response body is empty as documented.
- **Problem:** `PATCH /projects/:projectId` returns the updated project entity, but README documents an empty 200 response body. **File:** [src/projects/projects.controller.ts](src/projects/projects.controller.ts#L56-L64), [README.md](README.md#L84). **Fix:** Return `Promise<void>` so the response body is empty as documented.
- **Problem:** `POST /projects/:projectId/restore` returns the restored project entity, but README documents an empty 200 response body. **File:** [src/projects/projects.controller.ts](src/projects/projects.controller.ts#L75-L84), [README.md](README.md#L158). **Fix:** Return `Promise<void>`.
- **Problem:** All project endpoints leak `createdAt`, `updatedAt`, `deletedAt` columns from the raw `Project` entity — none of those are in the documented `{id, name, description, ownerId}` shape. **File:** [src/projects/projects.controller.ts](src/projects/projects.controller.ts#L37-L45), [src/projects/entities/project.entity.ts](src/projects/entities/project.entity.ts#L24-L31). **Fix:** Introduce a `ProjectResponseDto` (analogous to `UserResponseDto`) and return only the four documented fields.
- **Problem:** `GET /tickets`, `GET /tickets/:ticketId`, and `POST /tickets` return the raw `Ticket` entity which exposes `deletedByCascade` and `deletedAt` (and `version` on the array variants). None of those fields are in the documented response shape. **File:** [src/tickets/tickets.controller.ts](src/tickets/tickets.controller.ts#L106-L123), [src/tickets/entities/ticket.entity.ts](src/tickets/entities/ticket.entity.ts#L42-L58). **Fix:** Build a `TicketResponseDto` (used for arrays too) so internal columns aren't returned.
- **Problem:** `ETagInterceptor` explicitly skips arrays, so `version` leaks on `GET /tickets`, `GET /tickets/deleted`, `GET /tickets/:id/dependencies`, and `GET /tickets/:id/comments`. **File:** [src/common/interceptors/etag.interceptor.ts](src/common/interceptors/etag.interceptor.ts#L23-L37). **Fix:** Either strip `version` from each array element in the interceptor, or hide it via per-list response DTOs.
- **Problem:** `PATCH /tickets/:ticketId` returns the updated ticket entity, but README documents an empty 200 response body. **File:** [src/tickets/tickets.controller.ts](src/tickets/tickets.controller.ts#L125-L134), [README.md](README.md#L97). **Fix:** Return `Promise<void>` (still rely on the `ETag` header for the new version).
- **Problem:** `POST /tickets/:ticketId/restore` returns the restored ticket entity, but README documents an empty 200 response body. **File:** [src/tickets/tickets.controller.ts](src/tickets/tickets.controller.ts#L146-L155), [README.md](README.md#L151). **Fix:** Return `Promise<void>`.
- **Problem:** `GET /tickets/deleted` returns raw `Ticket[]` including `assigneeId`, `dueDate`, `isOverdue`, `version`, `deletedByCascade`, `deletedAt`, but README documents only `{id, title, status, priority, type, projectId}` for the deleted-tickets row. **File:** [src/tickets/tickets.controller.ts](src/tickets/tickets.controller.ts#L53-L60), [README.md](README.md#L150). **Fix:** Return a narrowed DTO matching the documented six-field shape.
- **Problem:** `POST /tickets/import` extracts `projectId` from `@Body('projectId', ParseIntPipe)` rather than a typed multipart-form DTO; the documented "form field" path lacks validation guardrails and bypasses `forbidNonWhitelisted`. **File:** [src/tickets/tickets.controller.ts](src/tickets/tickets.controller.ts#L87-L104). **Fix:** Add an `ImportTicketsDto` for the form fields and use `@Body() dto: ImportTicketsDto`.
- **Problem:** `GET /tickets/:id/comments` returns each item with `version`, `createdAt`, `updatedAt` — not in the documented `{id, ticketId, authorId, content, mentionedUsers}` shape. **File:** [src/comments/comments.controller.ts](src/comments/comments.controller.ts#L29-L32), [src/comments/comments.service.ts](src/comments/comments.service.ts#L320-L338). **Fix:** Build the list response without `version`/`createdAt`/`updatedAt`.
- **Problem:** README example response for `GET /audit-logs` shows `"action": "CREATE"` and bare `"entityType": "TICKET"`, but the implementation enum uses verb-prefixed forms like `TICKET_CREATE`. Clients filtering on `action=CREATE` per README always return empty. **File:** [src/common/enums/audit-action.enum.ts](src/common/enums/audit-action.enum.ts#L1-L31), [README.md](README.md#L119). **Fix:** Refactor `AuditAction` to the unprefixed verbs the README documents (`CREATE`, `UPDATE`, `DELETE`, `RESTORE`, `AUTO_ESCALATE`, `AUTO_ASSIGN`, `LOGIN`) and rely on the existing `entityType` column to disambiguate; update every emit site and the migration's `audit_logs_action_enum` accordingly.
- **Problem:** `entityId` query parameter on `/audit-logs` is parsed with `parseInt(entityId, 10)` and silently passed to the filter even when `NaN`, matching nothing instead of returning 400. **File:** [src/audit-log/audit-logs.controller.ts](src/audit-log/audit-logs.controller.ts#L14-L24). **Fix:** Use `ParseIntPipe` with `optional: true` (or guard with `Number.isInteger`) so a malformed query returns 400.
- **Problem:** `GET /users/:userId/mentions` does not validate that `:userId` references an existing user — the service runs the query directly and returns `{data:[], total:0, page}` for unknown ids instead of 404. **File:** [src/comments/mentions/mentions.controller.ts](src/comments/mentions/mentions.controller.ts#L8-L15), [src/comments/comments.service.ts](src/comments/comments.service.ts#L194-L235). **Fix:** Call `users.existsAndActive(userId)` first and throw 404 if missing.
- **Problem:** Mention items in `data[]` leak `version`, `createdAt`, `updatedAt` — not in the documented shape. **File:** [src/comments/comments.service.ts](src/comments/comments.service.ts#L320-L338), [README.md](README.md#L166). **Fix:** Build a response DTO that returns only `{id, ticketId, authorId, content, mentionedUsers}`.

## Requirements

- **Problem:** `DELETE /users/:userId` has no role guard, so any authenticated DEVELOPER can soft-delete any other user (R-USER-2). **File:** [src/users/users.controller.ts](src/users/users.controller.ts#L67-L74). **Fix:** Add `@UseGuards(RolesGuard)` + `@Roles(UserRole.ADMIN)` to the `delete` handler (and to `update`, since today a DEVELOPER can rename/demote any other DEVELOPER). **Test:** [test/users.e2e-spec.ts](test/users.e2e-spec.ts) — assert a DEVELOPER token receives 403 from `DELETE /users/:id`.
- **Problem:** `assertCanAssignRole` only blocks DEVELOPER → ADMIN escalation; it does not prevent a non-ADMIN actor from invoking PATCH at all (R-USER-3). The check lives in the controller rather than the service, so any non-HTTP caller bypasses it. **File:** [src/users/users.controller.ts](src/users/users.controller.ts#L58-L65). **Fix:** Move role-assignment authorisation into `UsersService.update`, gate the entire `role` field write on `actor.role === ADMIN`, and reject self-role-change to prevent privilege escalation.
- **Problem:** `CommentsService.delete` calls `this.comments.delete({ id })` which is TypeORM **hard** delete (R-COM-3) — the row is permanently removed, breaking the cascade-restore contract. **File:** [src/comments/comments.service.ts](src/comments/comments.service.ts#L152-L172). **Fix:** Replace with `await this.comments.softDelete(commentId)`; ensure `findForTicket` already filters `deletedAt IS NULL`. **Test:** [test/comments.e2e-spec.ts](test/comments.e2e-spec.ts) — after DELETE the row should still exist with `deletedAt` set and not appear in the list.
- **Problem:** `EscalationCron.handle` has no try/catch (R-AUDIT-1). If `runEscalation` throws mid-loop, the partial run loses subsequent audit rows without any visible signal. **File:** [src/tickets/escalation/escalation.cron.ts](src/tickets/escalation/escalation.cron.ts#L11-L15). **Fix:** Wrap the body in try/catch and `this.logger.error`.
- **Problem:** `GET /audit-logs` accepts `entityType`, `entityId`, `action`, `actor` as raw query strings — typed as enums but never validated (R-AUDIT-2 / R-ADD-1). `?actor=foo` is silently merged into the TypeORM `where` clause and matches nothing, instead of 400. **File:** [src/audit-log/audit-logs.controller.ts](src/audit-log/audit-logs.controller.ts#L12-L26). **Fix:** Define an `AuditLogQueryDto` with `@IsOptional() @IsEnum(...)` per field and `@IsInt() @Type(() => Number)` for `entityId`, then bind via `@Query() filter: AuditLogQueryDto`.
- **Problem:** `assertBlockersResolvedForDone` fetches blockers via `this.tickets.find({ where: { id: In(...) } })` with no `deletedAt: IsNull()` filter (R-DEP-3). A soft-deleted blocker counts as "still blocking" and refuses the DONE transition. **File:** [src/tickets/dependencies/dependencies.service.ts](src/tickets/dependencies/dependencies.service.ts#L109-L126). **Fix:** Add `deletedAt: IsNull()` to the `tickets.find` call. **Test:** [test/dependencies.e2e-spec.ts](test/dependencies.e2e-spec.ts) — soft-delete blocker, move ticket to DONE should succeed.
- **Problem:** `MAX_ATTACHMENT_SIZE` is enforced via multer `limits.fileSize` (R-ATT-1), but a `LIMIT_FILE_SIZE` error is surfaced by Nest as 500/413 with an opaque body unless an exception filter translates it; PDF §4.1 requires "clear error". **File:** [src/tickets/attachments/attachments.controller.ts](src/tickets/attachments/attachments.controller.ts#L36-L42). **Fix:** Register a `MulterError` exception filter or use `fileFilter` to translate the size error to `BadRequestException('Attachment exceeds 10 MB limit')`. **Test:** [test/attachments.e2e-spec.ts](test/attachments.e2e-spec.ts) — POST an 11 MB file, assert 4xx with the message.
- **Problem:** `FileTypeValidator` is created with `skipMagicNumbersValidation: true` (R-ATT-2), so the check trusts the client-supplied `Content-Type` header. A user can upload an arbitrary binary as `image/png` and pass validation. **File:** [src/tickets/attachments/attachments.controller.ts](src/tickets/attachments/attachments.controller.ts#L47-L52). **Fix:** Remove `skipMagicNumbersValidation` so the validator inspects the file signature, or add a manual magic-byte check on the buffer.
- **Problem:** CSV import `validate(dto, { whitelist: true })` lacks `forbidNonWhitelisted: true` (R-ADD-1), so CSV rows carrying extra columns are silently dropped instead of being reported in `errors[]`. **File:** [src/tickets/import-export/tickets-csv.service.ts](src/tickets/import-export/tickets-csv.service.ts#L111-L122). **Fix:** Pass `{ whitelist: true, forbidNonWhitelisted: true }` so unknown columns surface as row errors.
- **Problem:** `MENTION_REGEX = /@([a-zA-Z0-9_]+)/g` excludes hyphens and non-ASCII letters (R-MEN-1), so a user `@josé` or `@jane-doe` cannot be mentioned. **File:** [src/comments/mentions/mention-parser.ts](src/comments/mentions/mention-parser.ts#L5-L18). **Fix:** Broaden the regex to `/@([\p{L}\p{N}_\-]+)/gu`.
- **Problem:** On `toResponseWithLookup`, the per-comment user lookup is `userRepo.find({ where: { id: In(ids) } })` with no soft-delete filter (R-MEN-3 / R-USER-2 trade-off); soft-deleted users still surface in `mentionedUsers`. **File:** [src/comments/comments.service.ts](src/comments/comments.service.ts#L303-L318). **Fix:** Add `deletedAt: IsNull()` so the array omits soft-deleted users, matching the rest of the live-only read paths.
- **Problem:** When a HIGH overdue ticket is promoted to CRITICAL, `isOverdue` is NOT set on the same tick — the `else if` branch only enters when the ticket is already at CRITICAL (R-ESC-3 / R-ESC-5). The current impl. delays the flip until the next daily tick. **File:** [src/tickets/escalation/escalation.service.ts](src/tickets/escalation/escalation.service.ts#L42-L49). **Fix:** When the bumped priority lands at CRITICAL, also set `ticket.isOverdue = true` in the same iteration. **Test:** [test/escalation.e2e-spec.ts](test/escalation.e2e-spec.ts) — a HIGH overdue ticket should become `CRITICAL, isOverdue=true` after one tick.
- **Problem:** `UpdateTicketDto.assigneeId` is typed `number | null` and the service has a `dto.assigneeId !== null` branch (R-ASSIGN-7), but the DTO only annotates `@IsInt()`, so `class-validator` rejects an explicit `null` with 400. The "unassign" path is unreachable from the API. **File:** [src/tickets/dto/update-ticket.dto.ts](src/tickets/dto/update-ticket.dto.ts#L33-L36). **Fix:** Add `@ValidateIf((_, v) => v !== null)` to skip `@IsInt` when the body sends `null`. **Test:** [test/tickets.e2e-spec.ts](test/tickets.e2e-spec.ts) — PATCH `{ assigneeId: null }` should clear assignment, not 400.
